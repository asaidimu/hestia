## 2. Key assumptions that must change

### 2.1 Generator must distinguish "library's own autogen" from "downstream autogen"

**Problem:** The generator currently scans `internal/app/*/feature.go` and generates `collectFeatureRegistrations` that references `SystemModule` fields directly (`m.userModel`, `m.disp`, etc.). When hestia is a dependency, the binary cannot scan a library package's source files.

**Solution:** Two generator modes:

| Mode | When | What it generates |
|---|---|---|
| `hestia generate --self` | Run inside the hestia library repo (pre-commit) | Library's `internal/autogen/modules.go` ‚Äî includes system module's internal wiring |
| `hestia generate` (default) | Run in downstream project | Downstream's `internal/autogen/modules.go` ‚Äî imports system module from `github.com/asaidimu/hestia`, plus any user modules from `module_sources` |

### 2.2 `gen_features.go` must be subsumed by `autogen/modules.go`

**Problem:** `gen_features.go` generates a bespoke `collectFeatureRegistrations` function that hard-wires system module dependencies. This is not extensible.

**Solution:** The system module (in the hestia library) becomes self-wiring. Each feature's `feature.go` calls `Registrations(Dependencies{‚Ä¶})` inside its own module's `Setup()`, just like external modules already do. No more generated `collectFeatureRegistrations`. The only generated file is `internal/autogen/modules.go` which lists all discovered modules uniformly.

### 2.3 System module must use `module.go` naming convention

**Problem:** The system module lives at `internal/app/app.go`, but the convention is `module.go`.

**Solution:** Rename `internal/app/app.go` ‚Üí `internal/app/module.go` (in the hestia library). The scanner looks for `module.go` in `./[source]/[name]/` to identify a module.

### 2.4 `hestia.go` re-exports must not reference the downstream project

**Problem:** The draft `hestia.go` references `github.com/asaidimu/hestia/internal/*`. After extraction these become `github.com/asaidimu/hestia/internal/*`.

**Solution:** All imports in `hestia.go` point to `github.com/asaidimu/hestia/internal/‚Ä¶`. Downstream users only import `github.com/asaidimu/hestia`.

### 2.5 `hestia.json` config lives in the library too

The library ships its own `hestia.json`:

```json
{
  "module": "github.com/asaidimu/hestia",
  "module_sources": ["internal"],
  "module_target": "module",
  "cmds": []
}
```

The downstream project has its own `hestia.json`:

```json
{
  "module": "github.com/myorg/my-service",
  "module_sources": ["module"],
  "module_target": "module",
  "cmds": ["server"]
}
```

### 2.6 `cmd/scaffold` and `module scaffold` templates must reference the hestia library

Instead of `"github.com/asaidimu/hestia/internal/abstract"`, templates use `"github.com/asaidimu/hestia"`. The generated `feature.go` imports from the hestia library, not from the downstream project's internal packages.

## 3. Migration steps (server ‚Üí hestia library)

### Phase 1 ‚Äî Extract (in this repo)
1. Copy `internal/app/`, `internal/abstract/`, `internal/boot/`, `internal/core/`, `internal/interface/`, `migrations/` into the hestia library repo
2. Rename `internal/app/app.go` ‚Üí `internal/app/module.go`
3. Rewrite `hestia.go` to use `github.com/asaidimu/hestia/internal/‚Ä¶` imports
4. Add `--self` flag to the `generate` command for library-mode autogen
5. Ship a pre-committed `internal/autogen/modules.go` in the library

### Phase 2 ‚Äî Consume (in the downstream)
1. Add `github.com/asaidimu/hestia` as a dependency
2. Remove `internal/app/`, `internal/abstract/`, `internal/boot/`, `internal/core/`, `internal/interface/`, `migrations/` from the downstream project
3. Update `cmd/server/main.go` to use `hestia.Setup(cfg, ‚Ä¶)`
4. Update `hestia.json` to point to downstream-only `module_sources`
5. Run `hestia generate`

### Phase 3 ‚Äî Cleanup
1. Delete `cmd/hestia/` from the downstream project (users run the library's `hestia` binary, or run `go run github.com/asaidimu/hestia/cmd/hestia generate`)
2. `hestia scaffold cmd` creates a one-liner `main.go` that calls `hestia.Setup`
